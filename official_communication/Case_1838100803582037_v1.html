Title
Server-Side Authorization Bypass: Instagram Mobile Web Returns Private Timeline Data to Unauthenticated Requests
Description/Impact
Instagram's web interface contains a critical server-side authorization vulnerability that causes private account timeline data to be included in HTML responses sent to completely unauthenticated users. This is NOT a caching issue - Instagram's backend servers are actively generating and delivering private content to unauthorized requests.

THE VULNERABILITY:

When an unauthenticated user requests a private Instagram profile using mobile headers, Instagram's server responds with HTML that contains embedded JSON data including the private account's complete timeline with direct links to all private posts.

Key technical details:
- This occurs in the INITIAL server response (not cached data)
- The server embeds JSON inside <script type="application/json"> tags in the HTML
- This JSON contains the complete timeline structure: posts, images, metadata
- The vulnerability exists in Instagram's server-side rendering logic for mobile web
- No cookies, authentication tokens, or prior access required

WHY THIS IS NOT A CACHING ISSUE:

I understand your response might be: "CDN/cache limitations are expected behavior."

However, this is fundamentally different:

CACHING ISSUE (not a vulnerability):
- User previously had authorized access to content
- Content was legitimately delivered to their device
- Old cached URLs continue to work after access is revoked
- Expected limitation of CDN architecture

THIS VULNERABILITY (critical authorization bypass):
- User has NEVER had authorized access to the private account
- User has NEVER followed or viewed the account while logged in
- User makes a fresh HTTP request with NO authentication
- Instagram's SERVER actively generates and returns private timeline data
- This happens on EVERY request, not from cache

The server is incorrectly including private data in responses to unauthenticated clients.

TECHNICAL EXPLANATION:

REQUEST (from completely unauthenticated client):
- URL: <private_username>/
- Method: GET
- Cookies: NONE (fresh incognito browser)
- Authentication: NONE
- Headers: mentioned in file.

RESPONSE (from Instagram's server):
- HTML document containing embedded JSON
- JSON located in: <script type="application/json" data-content-len="...">
- JSON structure path: require → ScheduledServerJS → __bbox → RelayPrefetchedStreamCache → result → data → xig_user_by_igid_v2 → polaris_timeline_connection
- Contains: Complete array of private posts (edges → nodes)
- Each post includes: Post ID, image URLs (image_versions2), captions, metadata

EXPLOITATION:
- Parse the HTML response
- Extract JSON from script tag
- Navigate to polaris_timeline_connection object
- Extract image URLs from image_versions2 arrays
- URLs are directly accessible (CDN links)

This entire process requires ZERO authentication and works on ANY private account.

PROOF OF CONCEPT:

I have created a Python proof-of-concept script that demonstrates this vulnerability:

SCRIPT CONTENTS:
- Constructs HTTP request with mobile headers and NO authentication
- Sends GET request to private Instagram profile
- Receives HTML response from Instagram's server
- Parses HTML to extract embedded JSON script tags
- Locates polaris_timeline_connection structure in JSON
- Extracts all private post image URLs from image_versions2 arrays
- Decodes escaped URLs
- Saves accessible private content URLs to file

The script successfully extracts private post URLs from accounts I have never followed, never viewed, and have no relationship with.

- Created posts on my private account

SEVERITY: CRITICAL

PRIVACY VIOLATION:
- Complete circumvention of Instagram's private account privacy model
- Any person worldwide can access private posts from ANY private account
- Zero authentication, authorization, or follower relationship required
- Users have no way to detect unauthorized access to their private content

TECHNICAL IMPACT:

- Not a client-side or caching issue - this is a server authorization failure

Repro Steps
1. Prepare fresh environment. Open Chrome in Incognito/Private mode .Navigate to "Network" tab in DevTools

2. Decrease the viewport area by expanding the networks tab area.

3. In the browser address bar, navigate to: <private_username> Replace <private_username> with any private Instagram account username. The page will load showing "This Account is Private" message

4. In the Network tab, locate the request to /<private_username>. Click on this request to view details. Click the "Response" tab to see the HTML returned by Instagram's server. You are now viewing the HTML that Instagram's server generated and sent

5. In the Response view, press Ctrl+F to search. Search for: "polaris_timeline_connection". You will find this inside a <script type="application/json"> tag. This is NOT cached data - this is fresh JSON embedded by Instagram's server

6. Look at the JSON structure containing "polaris_timeline_connection". Navigate to: edges → [array of posts]. Each object in the edges array represents one private post. Look for: node → image_versions2 → candidates
This contains arrays of image URLs in different resolutions

7.
- Copy one of the URLs from the image_versions2 candidates array
- The URLs will be escaped (e.g., "https:\/\/scontent.cdninstagram.com\/...")
- Decode the URL (unescape the \/ characters and \uXXXX sequences)
- Paste the decoded URL in your browser address bar
- The private post image loads successfully WITHOUT any authentication
